## 三、项目经验详述

### 3.1 图片编辑器模块（imageEditor）

**目标**：实现一套前端图片编辑内核，支持「打开图片 → 选择编辑功能 → 实时预览 → 确认写入历史」的完整流程，数据与预览通过 Store + Stamp 机制解耦，编辑功能以独立子模块形式接入。

**架构与实现要点**：

- **状态与数据流**
  - 使用 Zustand 管理：当前基底图（`orgImageKey`）、历史列表（`historyList`）、历史索引（`historyIndex`）、当前编辑模块名（`currentModuleName`）、各模块参数（`currentModuleOptions`）
  - 通过 `optionsUpdateStamp` / `commitStamp` 控制「预览节奏」：控制面板只改 Store 参数，编辑器按 stamp 拉取最新参数执行请求或 Canvas 合成，避免滑块等高频操作导致每帧请求或卡顿
  - 预览结果缓存 `tempCurrentEditingImageCatchDataUrl` 用于双栏预览、抠图预览等
- **图像资源管理（Asset Pool）**
  - Store 仅存 `imageKey`（string），不存 Blob/ImageBitmap，减轻状态体积与序列化压力
  - 全局内存资源池：`putAsset(blob)` 返回 key，用于写入原图或历史项；各编辑器通过 `getCurrentImageKey()`、`useAssetBitmap(key)` 取 ImageBitmap 做 Canvas 渲染，或通过 `getCurrentImageUrl()`/`getCurrentImageBase64()` 做接口与合成
- **主框架业务（mainFrame.bis）**
  - 负责：根据 `currentImageKey` 从资源池取图、保存编辑结果为新 key 并写入历史、Undo/Redo、主容器 ResizeObserver 更新 `mainContainerSize`
  - 对外提供各类 `set*Action`（如 `setCurrentModuleOptionsAction`、`saveImageEditDataUrlAction`、`saveImageEditHistoryAction`、`undoImageEditHistoryAction`、`redoImageEditHistoryAction`）
  - 确认时执行 `saveImage()`：当前结果写历史并替换基底图；卸载时清理 `orgImageKey` 及历史中的 key
- **已实现的编辑子模块**
  - **起始页/预览**：载入图片、全图预览、缩放拖拽；选主菜单后进入对应编辑器
  - **裁剪（cut）**：矩形/圆角/圆形/自定义形状（SVG path）裁剪，翻转、旋转、缩放
  - **比例/尺寸（ratio）**：拉伸/裁剪/填充颜色，可调尺寸与比例
  - **旋转/镜头矫正（rotate）**：Y/X/鱼眼矫正，角度调节
  - **AI 消除笔（AIErase）**：画笔/矩形/套索，增加/减少消除区域，提交后走接口生成结果
  - **AI 超清（AISuperResolution）**：高清/AI 高清，提交后走代理接口生成超清图
  - **AI 自动抠图（aiAutoCutout）**：人像宠物/商品/图标等模式，获取 mask 后前端 Canvas 合成透明底；支持反选、单栏/双栏、小窗导航、原图栏半透明 mask 叠加
- **技术细节**
  - 画布区域根据 `currentModuleName` 动态加载对应编辑器（含裁剪、比例、旋转、AI 消除笔、AI 超清、AI 自动抠图），未选模块时展示 StartUI
  - 子菜单按 `currentMainMenu` 渲染「图片调整」或「抠图」面板，与当前编辑器一一对应
  - 各模块遵循：控制面板改 Store 参数（含 commitStamp）→ 编辑器监听 stamp 执行请求/合成 → 结果写入 `tempCurrentEditingImageCatchDataUrl` 或直接写历史；用户点击「应用/确认」时写入历史并替换基底图
  - 内部计算统一使用 px（固定 DPI 72），cm/in 仅用于 UI 显示换算

**成果**：形成可扩展的图片编辑内核，模块解耦、预览节奏可控、状态结构清晰，为后续 IndexedDB 持久化、Worker/OffscreenCanvas 加速等预留扩展空间。

### 3.2 创意专家聊天模块（creativityChat）

**目标**：提供创意专家场景下的对话与内容创作入口，支持按会话 ID 进入具体聊天界面。

**实现要点**：

- **路由与页面结构**
  - 主页面根据 `params.id` 判断：有 `creativityChatId` 时渲染「创意专家聊天」主界面，否则渲染起始页（StartPage），用于选择模式或创建新会话
  - 使用动态加载（dynamic import）拆分 StartPage 与 CreativityChat，减轻主 chunk 体积
- **创意专家聊天界面**
  - 布局：主内容区为 flex 横向，左侧为聊天框容器（ChatBox），右侧为聊天流（ChatStream）
  - 聊天框：支持文本输入、文件上传、示例选择等；与工作台 V2 共用部分 ChatBox 组件与 controllerhooks（如 chatBox.query、chatBox.bis）
  - 聊天流：展示当前会话的消息流，与后端/流式接口对接
- **状态与请求**
  - 通过 controllerhooks（如 chatBox、chatStreamList、conversationList）与 query 层对接接口，管理会话列表、当前会话消息、文件等
  - 使用 Zustand（如 chatStore）管理客户端会话与 UI 状态

**成果**：独立可用的创意专家对话模块，与工作台 V2 共享部分 UI 与数据逻辑，便于统一维护与体验一致。

### 3.3 工作台 V2（workBanchV2）

**目标**：提供脚本创作、混剪编辑、文件管理、对话协作的一体化工作台，支撑「脚本 → 视频生成 → 混剪」的流程。

**整体布局**：

- **顶部**（WorkBanchTop）：导航、标题、全局操作等
- **主内容区**（mainContentContainer）：
  - **左侧**：聊天流（WorkBanchLeft），展示当前会话的消息列表与流式输出
  - **中间**：文件管理（WorkBanchCenter），如可展开的文件列表、树形选择等
  - **右侧**：文件操作台（WorkBanchRight），脚本工作台或混剪工作台等
  - **聊天框**：可收起的 ChatBox，通过 `currentTCStyle.chatBoxStyle` 等做布局与动效
  - **展开文件管理器按钮**（FileListPartExBtn）、阴影/背景容器等

**核心子模块**：

- **脚本工作台（ScriptWorkShop）**
  - 用于编辑具体脚本内容，集成脚本编辑表格（ScriptEditorTable，动态加载）
  - 通过 `useScriptEditor` 等 bis 层与状态/接口对接，支持表格行数据（IDataRow）、视图编辑、文本编辑等
  - 提供返回等导航与加载态（ScriptEditorTableLoading）
- **混剪工作台（MixCutWorkShop）**
  - 左侧：标题、作者信息、头像等；集成视频播放器（VideoPlayer）、音量滑块等
  - 右侧：混剪配置（MixCutConf）、创作记录（CreationRecord）、混剪详情（MixCutDetail）等
  - 使用 WhiteTab 等组件做 Tab 切换与信息展示
- **通用 UI 与逻辑**
  - 组件：ChatBox、ChatStream、ButtonList、RadioButtonList、TreeDropDown、ExpandableFileList、ScriptEditorTable、VideoPlayer、MiniAIgenerator、DotPagination、SelectionList、TagDeleteableBtn、EditableTitle、FormText/FormFileuploader 等
  - 状态与业务：frame.bis（当前会话 ID、布局状态）、rightWorkBanch.bis、taskList.bis、conversationListV2.bis、scriptEditor.bis 等
  - 请求层：chatBox.query、chatStreamList.query、conversationQuery.query 等

**成果**：工作台 V2 成为脚本与混剪流程的主入口，与创意专家聊天、文件管理、对话流形成统一动线。

### 3.4 整体分辨率适配（resoProvider）

**目标**：在不改业务逻辑的前提下，让整站按设计稿比例在不同分辨率下缩放，并支持与主题的 UI 尺寸（小/中/大）联动。

**实现要点**：

- **技术方案**
  - 使用自研封装的 `@bobliao/reso-hook`：`useReso` 传入设计稿宽高（如 1400×900）、缩放模式（如 `EresoMode.AUTO`）、防抖时间、缩放限制（min/max width/height）、是否关联 devicePixelRatio 等
  - 将得到的 `reso`（含 `data.scriptStr`、`funcs`）通过 `resoContext.Provider` 注入整棵组件树，供子组件按需消费
- **当前单端适配配置**
  - 设计稿：1400×900，基准字体 16px，mode 为 AUTO，debounce 150ms
  - scaleLimit：启用，maxWidth 2000，minWidth 1440，minHeight 900
  - viewPort 为 auto，`is_relate_with_devicePixelRatio: true`
  - 将 `reso.funcs` 挂到 `window.current_reso`，便于非 React 或全局脚本使用；卸载时调用 `reso.funcs.distory()` 并清理
- **与主题联动**
  - 从 `useThememanagerDataContext()` 读取 `uiSize`（small/medium/large），在客户端挂载后设置基准字体（如 12/16/18），再传给 `useReso` 的 `fontSize`，实现「界面尺寸」与「分辨率缩放」一致
- **SSR/水合**
  - 在 `head` 中注入 `reso.data.scriptStr` 的 script 标签，使首屏在客户端水合前即可应用缩放；因服务端与客户端该脚本输出可能不一致，对 script 使用 `suppressHydrationWarning` 避免 React 水合报错
- **预留双端配置**
  - 代码中保留 `resoCondition`（queryList）：横屏 1440×900、竖屏 750×1080 等多端配置，当前未启用，便于后续 H5 或多端统一接入

**成果**：整站具备统一的设计稿级缩放与字体基准，并可与主题 UI 尺寸联动，提升大屏/小屏下的可读性与一致性。

### 3.5 色板系统整理与深色/浅色主题（globals.css + themeManager）

**目标**：统一全站颜色引用、便于换肤与维护，并实现深色/浅色两套主题。

**色板体系（globals.css）**：

- **基础色板**
  - 在 `:root` 下定义 `--cl-color-index-1` ～ `--cl-color-index-46` 等，每个变量附带注释（用途、对应设计稿场景），避免魔法数字与重复色值
  - 浅色主题下为亮色背景、深色文字、品牌紫/蓝等；深色主题下在 `.dark` 中重写同一批 index 变量，改为深底、浅字、高对比度
- **语义化颜色（V2.0 颜色表）**
  - 在基础色板之上再定义语义变量，例如：
    - 背景类：`--color-v2-page`、`--color-v2-surface`、`--color-v2-subtle`、`--color-v2-ai-card`、`--color-v2-mask`、`--color-v2-dropDown`、`--color-v2-section-header` 等，均引用 `var(--cl-color-index-*)`
    - 文本类：`--color-v2-primary`、`--color-v2-strong`、`--color-v2-text-secondary`、`--color-v2-muted`、`--color-v2-title`、`--color-v2-warning` 等
    - 品牌与按钮：`--color-brand-primary`、`--color-brand-accent`、`--color-button-primary-bg/fg`、`--color-outline-border`、`--color-ghost-hover-bg`、`--color-v2-disabled-bg/fg` 等
    - 边框与分割线：`--color-subtle`、`--color-strong`、`--color-tag`、`--color-warning-soft` 等
    - 图标与杂项：`--color-icon-primary`、`--color-icon-secondary`、`--color-block-muted-bg`、`--color-pagination-dot-active/inactive` 等
  - 注释中标明与旧版语义变量（如 `--color-bg-`*、`--color-text-*`）的对应关系，便于迁移与检索
  - 原则：业务与组件只引用语义变量，不直接写 `--cl-color-index-*`，便于后续换肤时只改色板或 `.dark` 即可
- **与 Tailwind / shadcn 对接**
  - 在 `@theme inline` 中把 `--color-`*、`--radius-*` 等映射到 Tailwind 的 `--color-*`、`--radius-*`，并保留 `@custom-variant dark`，使 `dark:` 与类名 `.dark` 生效
  - 动画、滚动条、图表色等也使用 CSS 变量，保证主题一致

**深色/浅色主题实现（themeManagerProvider）**：

- **状态与持久化**
  - 使用 `useLocalStorage` 持久化：`cl_isDarkMode`（boolean）、`cl_Uisize`（small/medium/large）
  - 通过 Context 提供：`isDarkMode`、`setIsDarkMode`、`uiSize`、`setUiSize`、`isMounted` 等
- **应用方式**
  - `useEffect` 中根据 `isDarkMode` 在 `document.documentElement` 上添加或移除 `dark` 类，使 `globals.css` 中 `.dark` 下的变量生效
  - 使用 Ant Design 的 `ConfigProvider` + `theme.darkAlgorithm` / `theme.defaultAlgorithm`，并与语义变量对接（如 `colorPrimary`、`colorBgContainer`、`colorBorder`、`colorText`、DatePicker 的 cell 背景等），保证 Antd 组件随主题切换

**成果**：全站颜色可维护、可换肤，深色/浅色与 UI 尺寸（小/中/大）统一由主题管理器与 resoProvider 联动，为后续多主题、多品牌预留空间。
