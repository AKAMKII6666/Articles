1. 这个单页 Dashboard 你会怎么拆页面层 / 业务组件 / 通用组件？
	考虑到Polaris为原子化组件库，库内组件繁多，故不一一展开流水帐式回答，我只列原则:
	通用组件：
	1.所有从Polaris过来的原始组件都做一层薄封装，作为基础组件中的元组件,命名和Polaris保持一致，以方便后续改样式或对行为一致性进行调整，通常来说基础组件放在/baseUiComponents/meta下。为什么要这么做，用于控制组件的兼容性风险，以及后续可能的需求升级。这里的组件并不是以字典形式把每个Polaris组件进行封装，而是用到一个封装一个。
	2.SlimVID基础组件：放在/baseUiComponents/下，根据目前 Dashboard的情况来看，dataCard组件、icon、table、button、pagenation 、summaryBlock都可以用Polaris中的组件进行封装，将布局样式重构至设计稿一致即可。比较特殊的是抽屉组件，这个可能Polaris中没有，不过应该能用现成的Polaris组件通过修改s-box组件相应的style来实现，比如设置position:fixed,width:30%,z-index:9等等。modal和upload组件Polaris均有现成的，可直接使用。

	基础组件的原则:
	1.不直接耦合appBridge
	2.不使用业务系统中任何单例模式里的状态和方法
	3.只通过参数来控制入参，事件响应，状态更改

2. 哪些状态放页面层，哪些放 store？
	1.store的原则：
		使用controllerHook来和store(zustandStore)、xhr接口、appBridge进行交互,向外提供业务状态和业务能力（functions/state）
		controllerHook通过参数调整两种运行模式：fetch:true | false
			fetch:true : 用于根页面，允许controllerHook主动调用数据获取一类接口。
			fetch:false : 用于业务组件，用于从controllerHook里拿取状态，允许controllerHook发送delete/modify/put/post的xhr但不允许get.对于业务组件而言，提供store中的状态，提供操作/功能函数。
	
	2.store的定义:
		globalStore -> globalStoreControllerHook
		dashboardStore -> dashboardControllerHook
		每个dashboardStore应该对应一个controllerHook
	
	3.位置:
		和jobs[]、pollMap、app是否启用、当前应用等级相关的状态/轮询业务逻辑/appBridge相关操作放globalStore/globalStoreControllerHook里
		原因：job轮询是系统级别的，应该在脱离了dashboard之后还在“后台”运行，这个后台应该就是指globalStore/globalStoreControllerHook,这样一来，就算用户在app中处理别的事情，也可以通过toast进行通知，Toast不用进store,只是让globalStoreControllerHook保持对appbridge的代理。

		OverviewCards、ActionSummary、RightDetailDrawer、UploadModal的业务逻辑和状态应（包括currentJobId）该放dashboardStore/dashboardControllerHook里,TopStatusBar / VideoTaskList：列表数据读 global jobs[]，展示与行事件可在 dashboard hook 包一层 不必再拆第三个 store。
		原因：这些都是dashboard的特性数据。
	
	4.责任:
		dashboardStore/dashboardControllerHook dashboard业务模块级别的状态、方法的单例模式
		globalStore/globalStoreControllerHook 系统级业务、状态的单例模式


	


3. Optimize 后这套 `job_id` 状态流你会怎么设计？
	前置条件:
		1.轮询接口是一个job一个xhr请求
		2.轮询入口为tasklist中点击“Optimize”
		3.轮询的状态(pollMap)和轮询业务都放在globalStore/globalStoreControllerHook里
		3.globalStore状态定义
			jobs[] - 商品列表（当前页）
			jobs[]CurrentPage - 当前页
			jobs[]PageCount - 每页显示多少条
			jobs[]TotalRow - 总行数
			pollMap - 当前轮询任务map
	
	全生命周期:
		启动：
		1.每条"任务"，应该是以商品为单位的，所以任务列表本质是商品列表，商品列表的每行数据下挂一个任务状态的子集。
		2.页面加载->加载商品列表(jobs[])，分页加载->根据当前页的商品去批量请求一次（GET /api/v1/jobs/{job_id}）接口，并将结果对应到jobs[]保存任务状态。将任务状态不为idle/succeeded/failed/cancelled的任务推送至pollMap里。
		3.启动一次pollMap数量检查，pollMap数量检查不为0 - 是否已经启动调度器 - 没有 - 启动，已经启动 - 继续。
		4.globalStoreControllerHook里有个调度器，用来控制一次发出的轮询xhr数量的。一次控制n个轮询xhr出现，例如5个。完成一个（succeeded/failed/cancelled），完成的那个从pollMap里去掉，更新到jobs[]里。如果不是完成，只是进度更新，没一次请求完成，就更新到jobs[]对应的商品里。如果更新时找不到就跳过，说民用户可能切换了页，但任务仍然保留。调度器里n个上限的轮询列表里少一个，就从pollMap里补上一个。轮询cd时间遵循prd稿子里的以后端返回时间为准( next_poll_after )。如果pollMap里已经没有任务，调度器就将现有任务全部完成。如果调度器里执行完一个任务，next()之前发现没有可执行任务。就停止调度器。

		调度器设计概要:
		使用单例 JobPollScheduler + pollMap。pollMap 仅存 in-flight 的 job_id 与 nextPollAt，不允许多 timer。调度器每 500ms tick 一次，在并发上限 5 内，对到期条目发起 GET /jobs/{job_id}，用返回的 next_poll_after 更新下次时间 succeeded/failed/cancelled 时 unregister 并 patch 列表行。pollMap 为空则停止调度器。首屏 GET /jobs 恢复所有进行中任务并 register submit 成功后 register 新 job。Toast 由 globalControllerHook 在终态触发。

		点击某一行商品的 Optimize:
		1.调用POST /api/v1/jobs/submit 获得jobid
		2.先使用jobid将任务乐观插入到pollMap里,调度器将自动运行。
		3.retry重新submit，用新 jobid。

		


4. 刷新页面后如何恢复状态？
	基础：job轮询是系统级别的，globalControllerhook里和globalstore进行交互，globalstore里存pollMap
	刷新后 GET /jobs，将 queued/processing 的 job 重新 register 进 pollMap，由调度器按 next_poll_after 继续轮询。
	globalControllerhook判断有没有正在执行的任务，有的话开轮询器，没有就不用开轮询器。
	重点:恢复的是服务端任务真相 + 重建轮询调度，不是恢复 iframe 里上次的前端内存。

	因为这个接口设计是一个任务一个轮询请求，那轮询请求可能最多会有无限个，容易产生网络堵塞，那我感觉可以做一个轮询管理器，将所有需要轮询的未完成任务从joblist抽出来，形成processing_list 然后送给一个请求器去处理，请求器一次只并发5个请求任务，空缺一个就从processing_list补一个，有成功或失败的任务，就从processing_list中移除。这样可以保证轮询更顺更可控。
	

5. App Embed 的安全替换逻辑你会怎么设计，才能不破坏原播放器行为？
	数据： 优化成功后由后端写入 Files + Metafield（持久化、与 Shopify 对齐）。Embed 运行时 V1 只走一条读通道：GET /api/v1/mapping?product_id=（可用 cache_buster），不做 Metafield 与 XHR 双读，避免双源不一致。Metafield 仅做记录，不作为前台必读。

	流程：

	1.根据当前商品页拿到 productid，请求 mapping 失败或 videos[] 为空 -> 静默 return，继续播原视频。
	2.只处理 mapping 里 status = active 的项 不按 productid 整页盲换。
	3.V1 只处理主视频位：在商品主内容区定位一个主 <video>（如 product media 区块内首个/主展示位），不重建、不插入、不删除 video 节点。
	4.读取 src / currentSrc / source[src] / data-src / data-video-url / data-source，对页面 URL 与 original_video_url 均做 normalizeUrl() 后再匹配 未命中则不动。
	5.播放前替换：可对优化 URL 先 preload，成功再改 src 与 poster 可先置空 src 避免先拉原片。失败 catch 回滚原地址，不黑屏、不闪断。
	6.不改 autoplay、controls、muted 等原播放器关键属性 播放中不扰动正在播放的实例。
	7.禁止（对齐 PRD）： 不插新 video、不替换节点结构、替换失败一律静默，顾客始终能回到 Shopify 原视频。

7.你认为这个 V1 前端落地里最容易踩坑的 3–5 个点是什么？你分别会怎么规避？
	风险：轮询泄漏 / 重复 timer
	规避：全局单例 JobPollScheduler + pollMap，禁止行组件 setInterval,pollMap 空则停调度器。

	风险：Embed 误替换（错视频、整页替换、黑屏）
	规避：只处理主 video,normalizeUrl + URL 精确匹配,不重建节点,preload 失败回滚,mapping 失败静默。

	风险：双数据源不一致（Metafield + mapping 双读）
	规避：V1 Embed 只读 mapping,写仍走 Metafield/Files。

	风险：主题差异导致找不到 video / 懒加载 src
	规避：兼容 data-src 等,播放前替换,V1 文档说明支持范围，失败即跳过。

	风险：刷新后状态丢失
	规避：以 GET /jobs 为真相重建 jobs[] 与 pollMap，不依赖 iframe 内存。

	风险：商品页前台请求 GET /api/v1/mapping?product_id= 可能遇跨域或后端难配
	规避：优先用 App Proxy / 同源方式暴露 mapping,若仍受阻，V1 可改为 Metafield + Liquid 单通道读取（仍避免双读 fallback）。

	风险: 据了解shopify推荐使用Polaris的默认样式进行页面编码，原因是为了样式统一，如果我们自己封装组件时样式过度自定义化可能有过审困难的风险。
	规避：布局结构，功能要求，元素视觉重要度，柑橘PRD稿子依照Polaris的设计规范优先做一版，不修改Polaris默认配色系统，圆角，间距等标准。后续再改成自己的风格不难。

function watchVideoFailure(video, originalSrc, onFail) {
  const cleanup = () => {
    video.removeEventListener('error', onError);
    clearTimeout(timer);
  };
  const onError = () => {
    cleanup();
    onFail(); // 回滚 originalSrc
  };
  // MediaError.code: 1=ABORTED 2=NETWORK 3=DECODE 4=SRC_NOT_SUPPORTED
  // 换 src 后若已 error：video.error !== null
  video.addEventListener('error', onError, { once: true });
}
