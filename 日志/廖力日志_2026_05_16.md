3.了解shopify saas工具的开发规范 - Polaris + appbridge
	1.Polaris组件的实现基于浏览器自定义标签，而非react组件的形式，Custom Elements /  Web Components ,所以在工程里可以看到react代码里并没有任何引用，因为Polaris的所有的组件来源于AppProvider文件中的一个js引用。ts支持则有单独的ts包来提供，不影响ts开发。所以我们可以直接在jsx中编写Polaris元素却不需要任何import.
	2.Polaris组件不像antd"服务到家"，形式上更像redix/shadcn  由一堆原子组件组成，在结合业务编写组件时，需要自己再组装一遍。
	3.这个站点的路由使用react SPA的文件路由，和nextjs类似但不完全一致，具体要看文档:https://reactrouter.com/how-to/file-route-conventions
	4.App Bridge的高频场景是toast,admin graphQl, 打开 Shopify 原生编辑器,选商品、选集合、选客户等（弹窗多选/单选）。,页面级 Loading,知道「谁在用什么环境」,当前用户 & App 授权范围

1. 这个单页 Dashboard 你会怎么拆页面层 / 业务组件 / 通用组件？

	（1）页面层
	单路由 DashboardPage：Page + 纵向拼装六块；负责 GET /jobs 初始化与恢复轮询入口；Drawer/Modal 挂页面级。不含轮询实现、不含 Embed、不含表格列业务细节。

	（2）业务层
	按 PRD 六模块拆分（各用 2～3 句：职责 + Polaris 结构 + 与设计稿对应点）；列表用 IndexTable；Drawer 突出 Original/Optimized 与 Timeline；Action 与 Overview 分工。

	（3）通用层
	MetricCard、StatusBadge、AppBanner 等薄封装 Polaris；原因：统一任务状态色、一级指标字号、后续改主题只改一处。

	提到这里尽量不要和appbridge产生任何交互：
	通用层 / 大部分业务展示组件：不负责 App Bridge，只负责 Polaris 展示与用户点击事件（通过 props 回调把动作交出去）。

	页面层或专门的 integration hook：在需要时调用 App Bridge（Toast、打开主题编辑器、Resource Picker 等）。

3. Optimize 后这套 `job_id` 状态流你会怎么设计？
	1.先轮询，后再 WebSocket 间隔跟 next_poll_after，不写死 5s
	2.全页只开一个 list 轮询器：当存在 queued/processing 时周期性 GET /jobs；没有进行中任务就停。强调不是「每行一个 timer」，而是 一个 scheduler，就是要做轮询生命周期管理。
	3.它这个轮询接口是单条任务的轮询接口，不是list,所以需要在拿到jobid之后在前端直接push到任务列表里，开启这个任务的轮询（乐观更新）。
	pollMap：同一 job_id 只允许一个 pending timer。
	Retry：新 POST /submit，新 job_id，旧 job 只读历史。
	Drawer 打开时：可对 currentJobId 加重 poll 或共用 pollMap 结果，避免再开一套。
	页面不可见：可降频或暂停 poll（加分，非必须）。

4. 刷新页面后如何恢复状态？
	基础：job轮询是系统级别的，globalControllerhook里和globalstore进行交互，globalstore里存job_list
	刷新页面后向服务器请求joblist，并请求轮询接口一次，取当前状态,globalControllerhook判断有没有正在执行的任务，有的话开轮询器，没有就不用开轮询器。
	重点:恢复的是服务端任务真相 + 重建轮询调度，不是恢复 iframe 里上次的前端内存。

	但我觉得还可以附加一个回答，你看是不是画蛇添足了，就是轮询管理器(轮询调度)，因为这个接口设计是一个任务一个轮询请求，那轮询请求可能最多会有无限个，容易产生网络堵塞，那我感觉可以做一个轮询管理器，将所有需要轮询的未完成任务从joblist抽出来，形成processing_list 然后送给一个请求器去处理，请求器一次只并发5个请求任务，空缺一个就从processing_list补一个，有成功或失败的任务，就从processing_list中移除。这样可以保证轮询更顺更可控。
